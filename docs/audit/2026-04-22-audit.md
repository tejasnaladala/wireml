**Date:** 2026-04-22
**Reviewers:** code-reviewer (frontend), python-reviewer (backend), security-reviewer (whole repo), + consistency pass.
**Scope:** Everything in `tejasnaladala/wireml` at `75b031e..1267138` on `main`, including the imported design system at `design-system/`.
**Local verification:** `pnpm -r typecheck` ✅, `pnpm --filter @wireml/web test` ✅ (3/3), `pnpm --filter @wireml/web build` ✅.

## Executive summary

- **5 CRITICAL** findings that either corrupt model outputs silently or open the runtime to attacker abuse. These should block any real use of the local runtime off-localhost and any "try the demo and train a model" flow.
- **13 HIGH** findings — product bugs and security gaps serious enough to address before announcing the repo publicly. Several directly contradict claims in the README.
- **13 MEDIUM** findings — UX regressions, architectural concerns, hardening gaps.
- **4 LOW / INFO** items — polish.

The repo builds green and the three shipped tests pass, but test coverage is thin and several correctness bugs slip through precisely because the affected paths are untested. The design system and prompt packs are strong on intent but import two CDN scripts without SRI and ship a sourcemapped production bundle.

The two findings to fix **before anyone clones this and trusts it with data**:
1. Topological sort in `packages/nodes/src/runner.ts` can execute the same node twice in diamond graphs (silent wrong outputs).
2. Full Python tracebacks are returned to HTTP clients on any `/runNode` error (info disclosure + invites CORS-abuse exploitation).

## CRITICAL (5)

### C1. `topoSort` double-executes nodes in diamond graphs — silent wrong outputs
**File:** `packages/nodes/src/runner.ts:58`
A node with two upstream predecessors can be pushed to `ready` twice because the `!sorted.includes(target)` check only prevents re-pushing *after* the target has been shifted into `sorted`. Between shift and process, a second predecessor can push the same node again. The diamond `A→B, A→C, B→D, C→D` triggers this today. Result: double-executed nodes mean duplicated side-effects (model downloads, training runs) and doubled features fed to array-input heads — silent incorrect predictions.
**Fix:** track pushed nodes in a `Set<string>` (`inReadyOrSorted`) and gate the push on it.

### C2. CLIP image runner returns one feature vector for N-image batches
**File:** `apps/web/src/runtime/WebGraphRunner.ts:95-101`
The code calls `pipe(images as any)` with a batch but wraps the result as `[Array.from(result.data)]` — a single-element array regardless of batch size. The linear head then trains on one example per node execution, with the label array zipped against the shortened feature array. Silent incorrect accuracy reports.
**Fix:** slice `result.data` into `images.length` equal-sized feature vectors (expected dim 512 for CLIP ViT-B/32). Add a contract test to `scheduler.test.ts` that builds a two-image mini-graph and asserts `features.length === 2`.

### C3. Full Python tracebacks returned in `/runNode` responses
**File:** `apps/runtime/wireml_runtime/main.py:76`, `wireml_runtime/schema.py:54`
`NodeError.stack = traceback.format_exc()` is serialized back to the client. Leaks absolute paths, torch/transformers versions, container layout, and any local-variable context that happened to appear in the traceback. Frontend mirrors the pattern at `apps/web/src/runtime/WebGraphRunner.ts:185` (err.stack → error.stack).
**Fix:** server-side `logger.exception(...)` to logs; drop `stack` from the response model entirely, or gate it behind `DEBUG=1` env var. Mirror the treatment on the frontend runner.

### C4. No request-body size limit + unbounded PIL decode
**File:** `apps/runtime/wireml_runtime/main.py` (no middleware), `wireml_runtime/nodes/backbones.py:22-26`
Uvicorn has no body limit configured and `PIL.Image.MAX_IMAGE_PIXELS` is never set. A base64-encoded decompression bomb or multi-GB body will happily make it through FastAPI and crash the runtime or exhaust memory.
**Fix:** add `Starlette ContentSizeLimitMiddleware` (cap 20 MB), set `Image.MAX_IMAGE_PIXELS = 25_000_000` at module init, and reject base64 payloads longer than the body cap before decode.

### C5. DirectML detection is written wrong — always falls through to CPU
**File:** `apps/runtime/wireml_runtime/device.py:88-96`
```python
importlib.import_module("onnxruntime.providers.DmlExecutionProvider")
```
`DmlExecutionProvider` is not a Python submodule — it's a provider string inside the onnxruntime C extension. Every Windows user falls through to CPU while the README claims DirectML support. The code silently lies.
**Fix:**
```python
import onnxruntime
return "DmlExecutionProvider" in onnxruntime.get_available_providers()
```

## HIGH (13)

### H1. Runtime exposes `:8787` on `0.0.0.0` with wildcard CORS
**Files:** `docker-compose.yml:21`, `apps/runtime/Dockerfile:34`, `apps/runtime/wireml_runtime/main.py:22-27`
Port is published to the host. CORS is `allow_origins=["*"]`, `allow_methods=["*"]`, `allow_headers=["*"]`. Any webpage the user visits can POST arbitrary node executions to the local runtime. Adds GPU-denial-of-service and inference-exfiltration attack surface.
**Fix:** bind `127.0.0.1:8787:8787` in compose, bind uvicorn to `127.0.0.1`, and lock CORS to `["http://localhost:5173", "http://localhost:8080"]` (configurable via env).

### H2. Deselection never clears the Inspector
**File:** `apps/web/src/canvas/Canvas.tsx:72`
`onNodesChange` only calls `select(ch.id)` when `ch.selected === true`. React Flow also emits `selected: false` on deselection; that path is ignored, so the Inspector shows stale data.
**Fix:** `if (ch.type === 'select') select(ch.selected ? ch.id : null);`

### H3. `moveNode` called on every drag tick → dirty churn
**File:** `apps/web/src/canvas/Canvas.tsx:68`
Every pixel of drag rewrites the nodes array and sets `dirty: true`. Becomes catastrophic with any future auto-save.
**Fix:** only commit position on `ch.dragging === false`, or debounce 60ms.

### H4. `onRun` captures stale `graph` via closure; no abort on re-run
**File:** `apps/web/src/App.tsx:48-59`
Editing the graph mid-run doesn't cancel in-flight execution. Running twice in a row can interleave results.
**Fix:** hold an `AbortController` in a ref; abort on re-run; the scheduler already accepts `signal`.

### H5. `getSchema` throws in render (NodeView + Inspector + scheduler)
**Files:** `apps/web/src/canvas/NodeView.tsx:30`, `apps/web/src/panels/Inspector.tsx:23`, `apps/web/src/runtime/scheduler.ts:54`
Loading a template that references an unknown `schemaId` (version skew, hand-edited JSON) blows up the entire canvas — no error boundary, no fallback.
**Fix:** make `getSchema` return `NodeSchema | undefined` and handle at call sites; render a red "unknown node" fallback in `NodeView`; wrap the canvas in an error boundary.

### H6. Duplicate edges accepted; corrupt `array: true` inputs
**File:** `apps/web/src/store/graphStore.ts:80-86`, `apps/web/src/canvas/Canvas.tsx:52-60`
`addEdge` doesn't dedupe by `(source.nodeId, source.port, target.nodeId, target.port)`. Rapid double-click / template import creates duplicates. Array-input aggregation in the scheduler then double-counts features.
**Fix:** dedupe in `addEdge`.

### H7. Model cache race under concurrent `/runNode` requests
**File:** `apps/runtime/wireml_runtime/nodes/backbones.py:17-18, 36-48`
Check-then-set with no lock. Two simultaneous loads of the same model will both download, both move to device, and leave a non-deterministic entry.
**Fix:** wrap each loader in `threading.Lock`, or collapse to `@functools.lru_cache`.

### H8. `to_torch_device` silently returns `cpu` for `mlx` and `directml`
**File:** `apps/runtime/wireml_runtime/device.py:145-157`
Apple users with MLX installed will transparently run models on CPU because the branch falls through. The runtime will report "power mode mlx" while silently using CPU.
**Fix:** raise `NotImplementedError` for `mlx`/`directml` in `to_torch_device` and thread MLX models through an MLX-aware runner, not a torch device.

### H9. `RuntimeCapabilities` response serializes `supported_nodes` (snake) not `supportedNodes` (camel)
**File:** `apps/runtime/wireml_runtime/schema.py:36`
Pydantic v2 with `populate_by_name` but without `by_alias` on response serialization emits the Python field name. Frontend at `packages/nodes/src/schema.ts:116` expects `supportedNodes` (camelCase). The contract is broken today — the web client silently reads `undefined` for the supported-node list and would fail any capability-gating that depended on it.
**Fix:** add `"by_alias": True` to `model_config` on all outbound models, or set `response_model_by_alias=True` on the endpoint.

### H10. lucide loaded from `unpkg.com/lucide@latest` without SRI
**File:** `design-system/ui_kits/workbench/index.html:43`
Three neighboring scripts have SRI hashes; lucide at `@latest` does not. Supply-chain single-point-of-failure.
**Fix:** pin to a specific version and add the matching `integrity="sha384-..."` attribute.

### H11. Production sourcemaps committed and deployed
**File:** `apps/web/vite.config.ts:25`
`sourcemap: true` emits full `.map` files in `dist/`, which the Pages workflow publishes. Reconstructs proprietary TS source from the public URL.
**Fix:** `sourcemap: 'hidden'` (keeps for error monitoring) or `false` for production. Ensure `dist/` stays ignored by git (currently is — good).

### H12. CI installs `uv` via unpinned remote script
**File:** `.github/workflows/ci.yml:36`
```yaml
run: curl -LsSf https://astral.sh/uv/install.sh | sh
```
Any compromise of `astral.sh` executes arbitrary code with `id-token: write`.
**Fix:** use the official `astral-sh/setup-uv@v4` GitHub Action, which pins by SHA.

### H13. `--frozen-lockfile=false` in all three pnpm installs
**Files:** `.github/workflows/ci.yml:22`, `.github/workflows/deploy-web.yml:35`, `apps/web/Dockerfile:10`
Explicitly disables lockfile integrity. Any floating resolution can silently change. Lockfile is present — just honor it.
**Fix:** drop the flag everywhere.

## What's good

This repo isn't a slop scaffold. Concretely:
- Clean package boundaries (`packages/nodes` is the single source of truth for schemas; both runners import it).
- Real tests (topological-sort has 3 meaningful tests; linear-head + confusion matrix are regression-tested with synthetic data).
- Design tokens consistently applied from `tailwind.config.js` through to the imported design system.
- No secrets, no hardcoded API keys, no third-party analytics embedded.
- Model IDs are not user-editable — avoids the most common Transformers.js supply-chain attack.
- Shipped green (`typecheck` / `test` / `build` all pass).
- Docs are grounded in the actual code, not aspirational — the README claims match the file tree.

The bugs flagged above are normal for a first-commit ambitious scaffold. None of them are architectural wrongs; every one is a localized fix.
